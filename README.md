Что такое охват

Тут три режима.

FunPay API
более узкий режим
старается применять UA ближе к запросам аккаунта/FunPay
Cardinal
более широкий runtime-режим
обновляет рабочие объекты и сессии процесса
Server
режим для Ubuntu/Linux
пишет текущий UA в env/profile текущего пользователя/сервиса
плюс применяет его в процессе
Логика между ними

если включаешь Server:
FunPay API выключается
Cardinal выключается
если включаешь FunPay API или Cardinal:
Server выключается
FunPay API и Cardinal могут быть включены вместе
Это специально сделано, чтобы не было конфликтной мешанины.

Нужно ли добавлять API?
Коротко: нет, отдельный пользовательский API ключ сюда не нужен.

Почему:

плагин использует бесплатный API WhatMyUserAgent
у него запрос выглядит так:
https://whatmyuseragent.com/api?ua=USERAGENT&key=NOTREQUIED
то есть отдельный ключ для пользователя не требуется
Источник:

WhatMyUserAgent Free API
Зачем тогда API вообще нужно
Не для генерации.
Не для ротации.
И не для “работы плагина в целом”.

Оно нужно только для проверки результата:

что распознаётся как браузер
что распознаётся как ОС
какой тип устройства видит внешний сервис
То есть API в этом плагине — это не зависимость, а валидатор.

Если хочешь, его можно вообще выключить, и плагин продолжит:

генерировать
ротировать
применять UA
Что важно понимать честно
Смена User-Agent:

может менять профиль HTTP-клиента
может делать запросы менее однотипными
может быть полезна для тестов, вариативности, имитации разных клиентов
Но она сама по себе не:

сбрасывает лимиты сайта
не обнуляет rate limit
не обходит ограничения по IP
не обходит ограничения по аккаунту
не “обнуляет API лимит” внешнего сервиса
Если лимит завязан на:

IP
аккаунт
cookie/session
fingerprint уровня выше, чем просто заголовок
то одна только смена User-Agent это не решит.

То есть использовать плагин надо с правильным ожиданием:

это инструмент управления User-Agent
а не волшебный сбросчик лимитов
